## 1. 项目概述

### 1.1 项目背景

SysML v2（Systems Modeling Language version 2）是 OMG（Object Management Group）发布的新一代系统建模语言标准。相较于 SysML v1.x，v2 版本在语法简洁性、语义精确性和工具互操作性方面有了显著提升。

当前市场上的 SysML v2 建模工具（如 SysON、Dassault Cameo 等）多为商业软件或基于 Eclipse 的复杂工具链，存在学习曲线陡峭、部署复杂、协作能力不足等问题。本项目旨在开发一款基于 Web 架构、Python 后端的轻量级图形化 SysML v2 建模工具，支持小团队协作，覆盖结构建模、行为建模、需求建模和参数建模等完整 MBSE（Model-Based Systems Engineering）能力。

### 1.2 项目目标

开发一款**图形化、可协作、跨阶段演进**的 SysML v2 系统建模软件，具备以下核心特征：

- 基于 SysML v2 标准文本格式（`.sysml2`）存储模型数据
- 纯图形拖拽式建模交互，零代码门槛
- Web 前端架构，Python 后端支撑
- 支持小团队（2-10 人）协作建模
- 覆盖完整的 MBSE 建模能力（结构、行为、需求、参数）
- 兼顾通用系统建模与国防武器装备设计领域

### 1.3 项目范围

| 维度 | 说明 |
|------|------|
| **建模标准** | OMG SysML v2（2024 正式版） |
| **建模能力** | 结构建模 + 行为建模 + 需求建模 + 参数建模（完整 MBSE） |
| **交互方式** | 纯图形拖拽（类似 Visio/Draw.io 体验） |
| **目标平台** | Windows（Anaconda Python 环境） |
| **架构模式** | Web 前端 + Python 后端（浏览器访问） |
| **开发模式** | 单人开发，无时间限制 |
| **许可证** | [待定] |

## 2. 目标用户与使用场景

### 2.1 用户角色

| 角色 | 权限 | 说明 |
|------|------|------|
| **一般用户** | 全部权限（创建、编辑、删除、导出、管理） | 团队核心成员 |
| **访客** | 只读（查看模型、导出视图） | 外部评审者、相关方 |

> 注：V1 阶段为单用户模式，V2 阶段引入用户认证与权限管理。

### 2.2 使用场景

1. **系统工程师**：对复杂系统进行结构化分解，定义部件、接口和连接关系
2. **需求工程师**：编写系统需求，建立需求间追溯关系，关联需求与设计元素
3. **设计团队**：在共享模型库上协同工作，各自负责不同子系统模块
4. **审阅者（访客）**：查看模型结构，导出图表用于文档或评审

### 2.3 应用领域

- **通用领域**：适用于各类工程系统的建模
- **国防武器装备设计**：支持武器系统的需求分析、架构设计、接口定义等

### 3.3 建模引导设计（参考 SysON 不足的改进）

针对 SysON 缺乏"模型视图的实例以及建模引导"的问题，V3 阶段提供：

- **建模模板库**：预置 BDD + IBD + 需求图的组合模板，用户选择后自动生成基础模型骨架
- **逐步建模向导**：引导用户按"需求 → 结构 → 行为 → 参数"的顺序逐步构建模型
- **实例视图**：基于部件定义自动生成部件实例树，展示实例间的连接关系

## 4. 非功能需求

### 4.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 画布渲染帧率 | ≥ 30 FPS | 100 个以下建模元素时 |
| 模型加载时间 | < 3 秒 | 1000 条模型语句以内 |
| 撤销/重做响应 | < 100ms | 即时反馈 |
| WebSocket 同步延迟 | < 500ms | V3 实时协同场景 |

### 4.2 可用性需求

- **上手时间**: 有 SysML 基础的用户 30 分钟内可完成首个模型创建
- **操作便捷**: 常用操作提供快捷键，工具箱分类清晰
- **中文支持**: 界面和文档支持中文（SysML v2 关键字保留英文）
- **错误提示**: 操作错误时给出明确的提示信息和修复建议

### 4.3 可靠性需求

- 自动保存（每 5 分钟或编辑暂停时触发）
- 崩溃恢复（重启后恢复未保存的编辑状态）
- `.sysml2` 文件合法性校验（保存前自动检查）

### 4.4 可维护性需求

- Python 后端代码遵循 PEP 8 规范
- 前端代码遵循 ESLint 标准配置
- 核心建模逻辑模块与 UI 模块解耦（便于单元测试和未来扩展）
- 使用 Git 进行版本管理

### 4.5 安全需求

| 需求 | V1 | V2 | V3 |
|------|:--:|:--:|:--:|
| 本地文件安全（不损坏原始文件） | ✓ | ✓ | ✓ |
| 密码哈希存储（bcrypt/argon2） | — | ✓ | ✓ |
| 用户会话管理（JWT） | — | ✓ | ✓ |
| API 访问控制 | — | ✓ | ✓ |
| HTTPS 传输加密 | — | ✓ | ✓ |

### 5.2 推荐技术选型

| 层次 | 技术 | 选型理由 |
|------|------|----------|
| **前端框架** | React 18+ | 生态最成熟，图形编辑器相关的库和方案最丰富（也可选 Vue 3，两者皆可胜任） |
| **图形渲染** | PixiJS | WebGL 渲染性能优异，比 Canvas 2D 库更适合大量建模元素的场景 |
| **连线库** | @pixi-essentials / 自研 | PixiJS 生态中的图形扩展；或基于 PixiJS 开发专用的连线组件 |
| **UI 组件库** | Ant Design 5 | 表格、树、表单等后台管理组件丰富，适合建模工具面板 |
| **状态管理** | Zustand | 轻量、无模板代码，适合图形编辑器高频状态变更 |
| **后端框架** | FastAPI | 异步原生支持，WebSocket 支持好，Python 生态兼容 |
| **语法解析** | Lark | Python 原生，EBNF 定义语法，比 ANTLR 更轻量 |
| **数据库** | SQLite (V2) → PostgreSQL (V3) | V2 用 SQLite 简化部署，V3 升级到 PostgreSQL 支持并发 |

### 5.3 SysML v2 文本解析核心

作为模型存储和交换的核心，需要实现 SysML v2 文本语法的解析器：

- **P0**: 解析器需完整支持 OMG SysML v2 标准文本语法（2024 正式版）
- **解析策略**: 使用 Lark 框架，基于 SysML v2 官方 EBNF 语法定义，生成 Python AST
- **双向转换**: `.sysml2` 文本 ↔ 内部模型对象（JSON）↔ 图形元素
- **增量更新**: 图形操作 → 局部模型变更 → 局部文本更新（非全量重写）

## 7. 风险与约束

### 7.1 技术风险

| 风险 | 等级 | 缓解措施 |
|------|:----:|----------|
| SysML v2 文本语法复杂度高，解析器开发困难 | **高** | 基于官方 EBNF 逐步实现，先覆盖常用语法子集；参考 SysON 等开源实现 |
| PixiJS 不适合传统图形编辑器（偏游戏引擎） | **中** | V1 初期做 PoC 验证，若不适合则切换到 Fabric.js 或 Konva.js |
| 实时协同（CRDT/OT）实现复杂 | **高** | V3 才实施，有充足的学习和调研时间 |
| XMI 交换规范不成熟 | **中** | V1 以 `.sysml2` 文本为主，XMI 只在 V2+ 阶段实现，且降级为基础支持 |
| 单人开发产能有限 | **中** | 严格分期，V1 聚焦核心闭环；善用开源库减少重复造轮子 |

### 7.2 约束条件

- **运行环境**: Windows + Anaconda Python（不跨 Linux/macOS）
- **开发模式**: 单人独立开发
- **建模规范**: 严格遵循 OMG SysML v2 2024 正式版标准
- **前端技术**: 浏览器访问，不依赖特定操作系统特性

### 7.3 依赖项

| 依赖 | 版本要求 | 用途 |
|------|----------|------|
| Python | ≥ 3.10 | 后端开发语言 |
| Node.js | ≥ 18 LTS | 前端开发环境 |
| Anaconda | 最新稳定版 | Python 环境管理 |
| SysML v2 规范 | 2024 正式版 | 语法和语义参考 |
